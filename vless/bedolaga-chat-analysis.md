# 📊 Анализ опыта операторов VPN (по материалам Bedolaga Social Club)

В этом документе собран глубокий анализ реального опыта, багов, конфигураций и обсуждений крупнейших администраторов VPN-сервисов в РФ из закрытого сообщества *Bedolaga Social Club*. Информация структурирована по ключевым направлениям: инфраструктура VPS/VDS хостингов, нюансы работы панели Remnawave, и обход режима «Белых списков» (БС).

---

## 1. 🏢 Анализ хостинг-провайдеров (VPS/Dedicated)

При построении VPN-инфраструктуры в РФ выбор хостинга критичен. Провайдеры делятся на три категории: транзитные (внутри РФ), зарубежные выходные ноды, и премиальные «мобильные» IP-адреса.

### 🥚 МТС Веб-Сервисы (MWS / VMware Cloud) — «Священный Грааль»
*   **В чем суть:** MWS предоставляет IP-адреса из подсетей мобильного оператора МТС/Yota (например, подсети `176.109.92.0/24`, `176.109.94.0/24`, `176.109.85.0/24`, а также приватная зона `176.109.82.0/24`).
*   **Преимущество:** ТСПУ/РКН не режет и не блокирует трафик, идущий на эти адреса, так как они определяются как опорная сеть мобильного оператора (чтобы не нарушить работу сотовой связи).
*   **Рынок:** Аккаунты MWS с большими квотами на IP активно скупаются администраторами (даже при блокировке консоли VMware, так как сами IP-адреса продолжают маршрутизироваться). Скорость портов на таких нодах достигает от 50 Мбит/с до 5 Гбит/с без лимита по трафику (оплата идет за полосу).

### 🖥️ Российские транзитные VPS
*   **Aeza (RU):** Ryzen 9 9950X, порт до 25 Гбит/с. Рекомендуемая новая подсеть: `83.147.255.0/24`. Геолокация может определяться с ошибками, но сам IP чистый.
    *   *Лайфхак сообщества:* Любые блокировки зарубежных ресурсов на транзитном сервере Aeza легко лечатся установкой **`warp-native`** (утилита Cloudflare Warp в режиме локального прокси), которая на Аезе встает стабильно с первого раза.
*   **Selectel:** Отличное качество сети и порты до 3 Гбит/с. Однако используется в основном для HAProxy-транзита, так как аренда дорогая, а трафик свыше 3 ТБ тарифицируется по 1.2 руб за ГБ.
*   **Datacheap (`vps.datacheap.ru`):** Популярный лоукостер для размещения транзитных нод. Отличается отсутствием жесткого шейпинга (ограничения) скорости под нагрузкой.
*   **VPSPay / Tube VDS:** Крайне дешевые серверы, но имеют плохую репутацию среди крупных админов. Наблюдается постоянный «шейв» (обрезание) скорости сети под нагрузкой и частые DDoS-атаки на инфраструктуру хостинга.
*   **QDE (`qde.ru`):** Провайдер, ранее популярный для размещения панелей управления. Зафиксирована крупная авария ("все умерло"), из-за чего админы потеряли базы данных (у многих бэкапы были старыми). Рекомендуется делать бэкапы БД панели ежедневно на внешние хранилища.

## 2. ⚡ Тонкости и баги Remnawave Panel & Nodes

Панель Remnawave активно развивается, но имеет ряд специфических багов и особенностей совместимости ядер Xray, которые необходимо учитывать при настройке.

### 🐛 Критический баг подсчета трафика Hysteria 2
*   **Проблема:** В нодах Remnawave версии **2.7.0** (использующих ядро Xray **26.3.27**) при подключении пользователей через протокол Hysteria 2 панель не отображает статус «Онлайн» и **не считает трафик** пользователя.
*   **Решение:** Ручное обновление ядра Xray в Docker-контейнере ноды до версии **26.6.1** или выше.
    *   *Инструкция:*
        1. Создать директорию: `mkdir -p /opt/remnanode/custom-xray && cd /opt/remnanode/custom-xray`
        2. Скачать свежий Xray-core: `wget https://github.com/XTLS/Xray-core/releases/download/v26.6.1/Xray-linux-64.zip`
        3. Распаковать: `unzip -o Xray-linux-64.zip`
        4. В `/opt/remnanode/docker-compose.yml` прокинуть новое ядро в контейнер:
           `- '/opt/remnanode/custom-xray/xray:/usr/local/bin/xray:ro'`
        5. Перезапустить ноду: `docker compose down && docker compose up -d`

### 🔄 Проблемы с миграцией и совместимостью клиентов (Happ / Incy)
*   **Happ (iOS) и новое ядро Xray:** После недавних обновлений ядра в приложении Happ у некоторых клиентов перестали работать xhttp-конфигурации при передаче в формате JSON, хотя стандартные GET-ссылки работают.
*   **Обратная совместимость PROXY протокола:** При настройке выходных нод параметр обратной совместимости `xver` (PROXY protocol) рекомендуется выставлять в значение `4` (или `0` если Caddy/Nginx не требуют передачи IP клиента), иначе старые версии клиентов Happ не смогут подключиться.
*   **Роутинг в Incy:** В отличие от Happ, в клиенте Incy передача кастомных правил маршрутизации (Routing Config) напрямую из панели Remnawave через подписку работает иначе. При возникновении утечек DNS на стороне Incy админы полностью вырезают блок `"dns"` из конфигурации Xray, пуская абсолютно весь DNS-трафик через туннель.

## 3. 🛰️ Детали настройки обхода БС (Белых Списков)

В условиях жестких блокировок CDN-фронтинг — единственный стабильный метод доставки трафика.

### 🐝 Beeline CDN — Фаворит для бесплатного трафика
*   **Почему Beeline CDN:** Билайн предоставляет безлимитный CDN-трафик на определенных типах аккаунтов. Это позволяет пропускать терабайты VPN-трафика через CDN Билайна бесплатно (готовые аккаунты с отлежкой продаются на черном рынке за 6–13$).
*   **Настройка:** Позволяет обходиться без постоянного поиска и перебора рабочих EDGE IP-адресов. Конфигурация работает стабильно как на iOS (Happ/Incy), так и на Android.

### 📱 Yandex Cloud CDN
*   **Стоимость:** Примерно 0.7 ₽ за 1 ГБ трафика.
*   **Лайфхак:** При регистрации нового аккаунта выдается приветственный грант в 4000 ₽. Этого гранта хватает примерно на 5-6 ТБ трафика (около месяца бесплатной работы CDN для небольшого проекта).
*   **Локальные проблемы:** CDN Яндекса стабилен, но в некоторых южных регионах (например, Краснодарский край) мобильные операторы местами блокируют или замедляют трафик к серверам Yandex Cloud CDN.

### 🛠️ Настройка xPadding в xhttp для обхода сигнатурного анализа
Для предотвращения блокировок CDN-трафика по объему пакетов (размеру фреймов) в транспорте `xhttp` обязательно настраивается обфускация размера пакетов (`xPadding`). Она добавляет случайный мусорный размер к заголовкам и телу HTTP-запроса.

*   **Пример обфускации через заголовок (X-Cache):**
    ```json
    "xhttpSettings": {
      "mode": "packet-up",
      "path": "/api-endpoint",
      "xPaddingObfsMode": true,
      "xPaddingKey": "секретный_ключ",
      "xPaddingHeader": "X-Cache",
      "xPaddingMethod": "tokenish",
      "xPaddingPlacement": "queryInHeader"
    }
    ```
*   **Пример обфускации для стандартного режима (auto):**
    ```json
    "xhttpSettings": {
      "mode": "auto",
      "host": "your-cdn.com",
      "path": "/stream",
      "extra": {
        "xPaddingBytes": "100-500" 
      }
    }
    ```
    *(Значение `xPaddingBytes` задает диапазон случайных байт, добавляемых к каждому пакету для разрушения паттерна шифрования).*

### 📶 Разделение трафика Wi-Fi и Мобильной связи
Поскольку CDN-трафик стоит денег (или требует дефицитных аккаунтов), администраторы настраивают клиенты так, чтобы через CDN шли только мобильные клиенты (где ТСПУ блокирует сильнее), а домашний Wi-Fi шел напрямую через обычный VLESS-Reality:
*   Используется блок **`burstObservatory`** в Xray для постоянного тестирования пинга. Если Reality-нода пингуется (например, на домашнем Wi-Fi провайдере блокировка слабее), клиент автоматически переключается на нее, минуя дорогой CDN.
